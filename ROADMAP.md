# CrewUp 프로젝트 로드맵

## 개요
- 목표: 테니스 동호회 관리 플랫폼 (MVP) → 다종목 동호회 → 범용 스포츠 커뮤니티
- 기술 스택: TypeScript, Expo (React Native + Web), Supabase, NativeWind
- 개발 환경: Expo CLI, Supabase CLI, VS Code
- 상세 요구사항: `docs/prd.md` 참조

## 진행 상태 범례
- ✅ 완료 | 🔄 진행 중 | 📋 예정 | ⏸️ 보류

## 프로젝트 현황 대시보드
- 전체 진행률: Phase 2/11
- 현재 Phase: Phase 2 (동호회 관리)
- 완료된 스프린트: Sprint 0, Sprint 1, Sprint 2
- 진행 중 스프린트: —
- 다음 스프린트: Sprint 3 (코드 품질 수정 + 가입/멤버 관리)
- 다음 마일스톤: Phase 2 완료 (동호회 생성/탐색/가입/멤버 관리)

## MVP 타임라인 추정

> 1인 바이브 코딩 기준, 1 Sprint = 1~3일 추정. 실제 진행 시 조정.

| Phase | Sprint | 예상 기간 | 누적 |
|-------|--------|----------|------|
| Phase 0: 프로젝트 기반 | Sprint 0 | ✅ 완료 | 1일 |
| Phase 1: 인증 | Sprint 1 | ✅ 완료 | 2일 |
| Phase 2: 동호회 관리 | Sprint 2~3 | 3~5일 | ~7일 |
| Phase 3: 일정 + 참석 + 자동등록 | Sprint 4~6 | 5~8일 | ~15일 |
| Phase 4: 알림 | Sprint 7 | 3~4일 | ~19일 |
| Phase 5: MVP 완성 | Sprint 8~9 | 5~7일 | ~26일 |

- MVP 출시까지 약 **4주** (바이브 코딩 풀타임 기준)
- 검토/피드백/수정 포함하면 **5~6주** 예상

## MVP 제외 범위 (Won't-have)

> MVP에서 의도적으로 제외하는 기능. 범위 흐림 방지를 위해 명시한다.

| 기능 | 제외 이유 | 예정 |
|------|----------|------|
| 결제 (충전/참가비/출금) | 사업자 등록 필요, MVP는 수동 입금 안내 | Phase 9 |
| 지도 연동 (장소 지도 표시) | 지도 SDK 연동 복잡, MVP는 주소 텍스트만 | Phase 11 |
| 다종목 (축구, 골프 등) | MVP는 테니스 집중, DB 구조만 확장 대비 | Phase 10 |
| 인력시장 다종목 확장 | MVP는 테니스 인력시장만, 다종목은 Phase 10 이후 | Phase 10+ |
| 카카오 알림톡 | 비용 + 사업자 필요, FCM으로 충분 | Phase 7 이후 |
| 채팅 기능 | 카톡 단톡방이 이미 역할 수행 | 미정 |
| 다국어 지원 | 한국 시장 전용 서비스 | 안 함 |
| 테마/다크모드 | 향후 테마 선택 기능으로 제공 | Phase 6 이후 |
| 다시설 자동 일정 (5곳 이후 추가) | Phase 7에서 인기 5곳까지 확장, 이후 추가는 수요 기반 | Phase 7 이후 |

## 기술 아키텍처 결정 사항

| 결정 | 선택 | 이유 |
|------|------|------|
| 프레임워크 | Expo (React Native) | 웹+앱 단일 코드, 바이브 코딩 최적 |
| 백엔드 | Supabase | Auth+DB+Realtime+Storage 올인원, 무료 티어 |
| 배포 (MVP) | Vercel (웹 PWA) | 0원, 인스타 링크 유입 최적 |
| 스타일링 | NativeWind | Tailwind 문법, 반응형, 웹/네이티브 통일 |
| 상태 관리 | Zustand + React Query | 클라이언트/서버 상태 분리 |
| 알림 | FCM (expo-notifications) | 무료, 전 플랫폼 |
| 소셜 로그인 | Supabase Auth (Google/카카오, 네이버 예정) | 무료, 서버 불필요 |

## 의존성 맵

```
Phase 0 (기반)
  └→ Phase 1 (인증)
       └→ Phase 2 (동호회)
            └→ Phase 3 (일정+참석+자동등록 — 목동 1곳)
                 └→ Phase 4 (알림)
                      └→ Phase 5 (MVP 완성+배포)
                           ├→ Phase 6 (앱 스토어)
                           └→ Phase 7 (자동 일정 확장 — 인기 5곳)
                                └→ Phase 8 (인력시장 — 테니스 픽업)
                                     └→ Phase 9 (결제)
                                          └→ Phase 10 (다종목 확장)
                                               └→ Phase 11 (지역+커뮤니티)
```

- 동호회 생성 → 목록 표시 → 타 유저 가입 신청 → 승인 → 멤버 목록 전체 플로우

- 데이터: React Query로 캐싱 + 무효화

## Phase 3: 일정 + 참석 + 자동 일정 등록 📋

> **마일스톤**: "일정을 잡고, 사람들이 참석/대기하며, 코트 예약이 자동으로 일정에 등록된다"

### 목표
캘린더 기반 일정관리, 참석/대기 시스템, 외부 시설 예약 크롤링을 통한 자동 일정 등록을 구축한다. 동호회 활동의 핵심 플로우.

### 작업 목록
#### Sprint 4: 캘린더 + 일정 CRUD
- Supabase 마이그레이션: `schedule_events`, `event_attendees`, `venues` + RLS
- 월별 캘린더 컴포넌트 (날짜 셀 + 일정 카드)
- 캘린더 월 이동, 오늘 강조, 지난 일정 흐리게
- 일정 생성 화면 (날짜, 시간, 장소, 정원, 참가비, 메모, 옵션)
- 자주 쓰는 장소 관리 (venues CRUD)
- 일정 상세 화면
- 일정 수정/취소 (Owner/Admin)

#### Sprint 5: 참석/대기 시스템
- 참석 신청 API (정원 체크 → attending 또는 waiting)
- 대기 순번 관리 (position)
- 참석 취소 → 대기 1번 자동 승격 (DB 트랜잭션 또는 Edge Function)
- 참석/대기/취소 UI (버튼 상태 전환)
- 참석자/대기자 명단 표시
- 일정 자동 마감 (참석자 = 정원 → closed)
- 참가비 입금 안내 (계좌 표시 + 복사 — 수동)
- Realtime 구독 (참석 현황 실시간 반영)

#### Sprint 6: 자동 일정 등록 (시설 예약 크롤링)
- Supabase 마이그레이션: `facility_credentials` 테이블 (시설 로그인 정보 암호화 저장) + RLS
- 시설 계정 연동 UI (동호회 관리 → "외부 시설 연동" — MVP는 목동테니스장만)
- Edge Function: 목동테니스장 예약 시스템 크롤링 (로그인 → 예약 목록 조회 → 파싱)
- 파싱 데이터: 예약 날짜, 시간대, 코트번호, 예약자명
- 파싱 결과 → `schedule_events` 자동 INSERT (중복 방지 로직 포함)
- 크롤링 주기 설정 (pg_cron 또는 Edge Function Cron — 1일 1~2회)
- 자동 등록된 일정에 "자동 등록" 배지 표시
- 크롤링 결과 로그/알림 (성공/실패/신규 일정 N건 등록)

### 완료 기준
- 캘린더에서 일정 생성 → 카드 표시 → 참석 → 마감 → 대기 → 취소 → 승격 전체 플로우
- 동시 참석 시 race condition 없음 (정원 초과 방지)
- Realtime: 다른 유저가 참석하면 내 화면에 즉시 반영
- 모바일 캘린더 터치 UX 원활
- 목동테니스장 계정 연동 → 예약 크롤링 → 일정 자동 등록 플로우 정상
- 중복 일정 방지 (이미 등록된 예약은 재등록하지 않음)

### 기술 고려사항
- 캘린더: 커스텀 구현 (목업 느낌 참고)
- 승격 로직: Edge Function 또는 PostgreSQL RPC (트랜잭션 필수)
- Realtime: `event_attendees` 변경 구독 → React Query 캐시 무효화
- 반응형: 캘린더 (모바일: 컴팩트, 데스크톱: 넓은 그리드)
- 크롤링: Supabase Edge Function (Deno) — HTTP 요청 + HTML 파싱 (cheerio/dom-parser)
- 자격증명 보안: 암호화 저장 (pgcrypto 또는 Vault), 서비스 키로만 복호화
- 확장성: 크롤러를 시설별 어댑터 패턴으로 설계 → 향후 다른 시설 추가 시 어댑터만 구현
- MVP: 목동테니스장만 지원. 시설 선택 UI는 있되 "목동테니스장"만 활성화

## Phase 8: 인력시장 (테니스 픽업 게임) ⏸️

> **마일스톤**: "코트 잡은 사람이 올리면, 개인이 자유롭게 참여하고, 노쇼는 추적된다"
> **배경**: 현재 카카오톡 단톡방(~300명)에서 운영 중인 시스템을 앱으로 대체

### 목표
동호회 가입 없이 개인이 코트를 올리고 참여하는 단발성 픽업 게임 마켓플레이스. 노쇼 관리, 참여 통계, 대량 일정 필터링이 핵심.

### 동호회 모델과의 차이점
| 항목 | 동호회 | 인력시장 |
|------|--------|----------|
| 소속 | 가입/멤버십 | 없음 (자유 참여) |
| 일정 성격 | 정기/반복 | 단발성 |
| 운영 | 클럽장/운영진 | 플랫폼 운영자 |
| 노쇼 | 동호회 내부 처리 | 플랫폼 차원 제재 |
| 규모 | 10~50명 | 300명+ |
| 일정 밀도 | 주 2~3회 | 하루 수십 건 |

### 작업 목록
#### Sprint 13: 픽업 게임 기반 + 코트 올리기
- Supabase 마이그레이션: `pickup_games`, `pickup_participants`, `user_penalties` 테이블 + RLS
- 코트 올리기 화면 (날짜, 시간, 테니스장, 코트번호, 정원, 참가비, 메모)
- 픽업 게임 목록 (캘린더 뷰 + 리스트 뷰)
- 캘린더 대량 일정 처리: 테니스장별 필터, 시간대 필터 (하루 수십 건 대응)
- 참여 신청/취소 UI
- 참여자 명단 표시

#### Sprint 14: 노쇼 관리 + 통계 + 운영 도구
- 운영자 대시보드 (노쇼 신고/확인 → 제재 처리)
- 노쇼 기록 + 자동 제재 정책 (N회 노쇼 → 참여 제한)
- 사용자 신뢰도 점수 (참여 횟수, 노쇼 비율, 참여율)
- 참여 통계 (월별 참여 횟수, 선호 테니스장, 선호 시간대)
- 사용자 프로필에 픽업 통계 표시
- 블랙리스트/제재 해제 관리

### 완료 기준
- 코트 올리기 → 참여 신청 → 게임 진행 → 노쇼 신고 → 제재 전체 플로우
- 하루 30건+ 일정에서 필터링으로 원하는 코트를 빠르게 찾을 수 있음
- 노쇼 3회 → 자동 참여 제한 동작
- 운영자가 제재 현황을 한눈에 파악 가능

### 기술 고려사항
- 동호회 시스템과 DB/UI 분리하되, 인증/프로필/알림은 공유
- 캘린더 대량 데이터: 가상화 + 서버사이드 필터링 (테니스장 ID, 날짜 범위)
- 노쇼 판정: 운영자 수동 확인 (자동 판정은 향후)
- 알림 연동: 참여한 게임 리마인더, 노쇼 경고 알림
