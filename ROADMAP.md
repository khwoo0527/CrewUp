# CrewUp 프로젝트 로드맵

## 개요
- 목표: 테니스 동호회 관리 플랫폼 (MVP) → 다종목 동호회 → 범용 스포츠 커뮤니티
- 기술 스택: TypeScript, Expo (React Native + Web), Supabase, NativeWind
- 개발 환경: Expo CLI, Supabase CLI, VS Code
- 상세 요구사항: `docs/prd.md` 참조

## 진행 상태 범례
- ✅ 완료 | 🔄 진행 중 | 📋 예정 | ⏸️ 보류

## 프로젝트 현황 대시보드
- 전체 진행률: Phase 0/10
- 현재 Phase: 시작 전
- 완료된 스프린트: 없음
- 다음 마일스톤: Phase 0 완료 (프로젝트 기반 구축)

## MVP 타임라인 추정

> 1인 바이브 코딩 기준, 1 Sprint = 1~3일 추정. 실제 진행 시 조정.

| Phase | Sprint | 예상 기간 | 누적 |
|-------|--------|----------|------|
| Phase 0: 프로젝트 기반 | Sprint 0 | 2~3일 | ~3일 |
| Phase 1: 인증 | Sprint 1 | 2~3일 | ~6일 |
| Phase 2: 동호회 관리 | Sprint 2~3 | 3~5일 | ~11일 |
| Phase 3: 일정 + 참석 | Sprint 4~5 | 4~6일 | ~17일 |
| Phase 4: 알림 | Sprint 6 | 3~4일 | ~21일 |
| Phase 5: MVP 완성 | Sprint 7~8 | 5~7일 | ~28일 |

- MVP 출시까지 약 **4주** (바이브 코딩 풀타임 기준)
- 검토/피드백/수정 포함하면 **5~6주** 예상

## MVP 제외 범위 (Won't-have)

> MVP에서 의도적으로 제외하는 기능. 범위 흐림 방지를 위해 명시한다.

| 기능 | 제외 이유 | 예정 |
|------|----------|------|
| 결제 (충전/참가비/출금) | 사업자 등록 필요, MVP는 수동 입금 안내 | Phase 7 |
| 지도 연동 (장소 지도 표시) | 지도 SDK 연동 복잡, MVP는 주소 텍스트만 | Phase 9 |
| 다종목 (축구, 골프 등) | MVP는 테니스 집중, DB 구조만 확장 대비 | Phase 8 |
| 개인 매칭/인력시장 | 플랫폼 확장 단계 | Phase 10 |
| 카카오 알림톡 | 비용 + 사업자 필요, FCM으로 충분 | Phase 7 이후 |
| 채팅 기능 | 카톡 단톡방이 이미 역할 수행 | 미정 |
| 다국어 지원 | 한국 시장 전용 서비스 | 안 함 |
| 테마/다크모드 | 향후 테마 선택 기능으로 제공 | Phase 6 이후 |

## 기술 아키텍처 결정 사항

| 결정 | 선택 | 이유 |
|------|------|------|
| 프레임워크 | Expo (React Native) | 웹+앱 단일 코드, 바이브 코딩 최적 |
| 백엔드 | Supabase | Auth+DB+Realtime+Storage 올인원, 무료 티어 |
| 배포 (MVP) | Vercel (웹 PWA) | 0원, 인스타 링크 유입 최적 |
| 스타일링 | NativeWind | Tailwind 문법, 반응형, 웹/네이티브 통일 |
| 상태 관리 | Zustand + React Query | 클라이언트/서버 상태 분리 |
| 알림 | FCM (expo-notifications) | 무료, 전 플랫폼 |
| 소셜 로그인 | Supabase Auth (카카오/네이버) | 무료, 서버 불필요 |

## 의존성 맵

```
Phase 0 (기반)
  └→ Phase 1 (인증)
       └→ Phase 2 (동호회)
            └→ Phase 3 (일정+참석)
                 └→ Phase 4 (알림)
                      └→ Phase 5 (MVP 완성+배포)
                           ├→ Phase 6 (앱 스토어) ─→ Phase 8 (다종목)
                           └→ Phase 7 (결제)           └→ Phase 9 (지역)
                                                            └→ Phase 10 (커뮤니티)
```

### 작업 목록
#### Sprint 0: 프로젝트 초기 설정

- 기존 HTML 목업 → `docs/mockup/`으로 이동
- Expo Router 파일 기반 라우팅 구조 설정
- NativeWind + Tailwind 설정

- 환경 변수 설정 (`.env.local`)
- Supabase 로컬 개발 환경 (`supabase init`)
- 기본 네비게이션 골격 (탭 바 + 스택)
- 공통 컴포넌트 기반 (Button, Card, Badge, Container, LoadingSpinner)

- ESLint + Prettier 설정

- Supabase 로컬 연결 확인

- NativeWind v4 + Tailwind CSS v3
- 모바일 퍼스트 반응형 기반을 이 Phase에서 확립
- `app/`은 라우팅만, `src/features/`에 비즈니스 로직

## Phase 1: 인증 시스템 📋

> **마일스톤**: "사용자가 로그인하고 프로필을 만든다"

### 목표
카카오/네이버 소셜 로그인, 온보딩(프로필 설정), 세션 관리를 구축한다. 이후 모든 기능의 사용자 식별 기반.

### 작업 목록
#### Sprint 1: 소셜 로그인 + 온보딩
- Supabase 마이그레이션: `profiles` 테이블 + RLS
- 카카오 OAuth 연동 (Supabase Auth)
- 네이버 OAuth 연동 (Supabase Auth)
- 로그인 화면 (소셜 로그인 버튼)
- 온보딩 화면 (닉네임, 프로필 사진, 활동 지역)
- `useAuth` 훅 (세션 관리, 로그인 상태 감지)
- 인증 가드 (`_layout.tsx` — 미로그인 시 로그인 화면으로)
- 프로필 수정 화면
- 로그아웃

### 완료 기준
- 카카오/네이버 → 온보딩 → 메인 화면 전체 플로우 정상
- 앱 재시작 시 세션 유지
- 미로그인 → 보호된 화면 접근 시 로그인으로 리다이렉트

### 기술 고려사항

- 프로필 이미지: `expo-image-picker` + Supabase Storage

## Phase 2: 동호회 관리 📋

> **마일스톤**: "동호회를 만들고, 찾고, 가입하고, 멤버를 관리한다"

### 목표
동호회의 전체 생명주기(생성 → 탐색 → 가입 → 멤버 관리)를 구축한다. 이 Phase가 끝나면 "사람이 모이는" 구조가 완성된다.

### 작업 목록
#### Sprint 2: DB 스키마 + 동호회 CRUD
- Supabase 마이그레이션: `sport_categories`, `clubs`, `club_members`, `join_requests` + RLS
- 시드 데이터 (sport_categories에 테니스)
- `supabase gen types` → DB 타입 생성
- 동호회 생성 화면 + API
- 동호회 목록 화면 (카드 그리드, 검색)
- 동호회 상세/소개 화면 (비회원용)
- 내 동호회 목록 (가입된 동호회 전환)

#### Sprint 3: 가입 + 멤버 관리
- 가입 신청 화면 + API (인사말 입력)
- 가입 대기 관리 (승인/거절 — Owner/Admin)
- 멤버 목록 화면 (역할, 가입일)
- 역할 변경 + 강퇴 (Owner/Admin)
- 동호회 관리 페이지 (정보 수정, 운영진 지정 — Owner/Admin 전용)
- 동호회 내부 접근 제어 (RLS: 멤버만)

### 완료 기준
- 동호회 생성 → 목록 표시 → 타 유저 가입 신청 → 승인 → 멤버 목록 전체 플로우
- 비회원: 목록/소개만 접근, 내부 콘텐츠 잠금
- 멤버: 동호회 내부 접근 가능
- Owner/Admin: 관리 페이지 접근 가능, Member: 불가
- 반응형: 카드 목록 (모바일 1열, 데스크톱 3열)

### 기술 고려사항
- 검색: Supabase `.ilike()` 또는 Full Text Search
- 리스트: 동호회 10개+ → FlashList
- 이미지: `expo-image` (동호회 대표 이미지)
- 데이터: React Query로 캐싱 + 무효화

## Phase 3: 일정 + 참석 시스템 📋

> **마일스톤**: "일정을 잡고, 사람들이 참석/대기한다"

### 목표
캘린더 기반 일정관리와 참석/대기 시스템을 구축한다. 동호회 활동의 핵심 플로우.

### 작업 목록
#### Sprint 4: 캘린더 + 일정 CRUD
- Supabase 마이그레이션: `schedule_events`, `event_attendees`, `venues` + RLS
- 월별 캘린더 컴포넌트 (날짜 셀 + 일정 카드)
- 캘린더 월 이동, 오늘 강조, 지난 일정 흐리게
- 일정 생성 화면 (날짜, 시간, 장소, 정원, 참가비, 메모, 옵션)
- 자주 쓰는 장소 관리 (venues CRUD)
- 일정 상세 화면
- 일정 수정/취소 (Owner/Admin)

#### Sprint 5: 참석/대기 시스템
- 참석 신청 API (정원 체크 → attending 또는 waiting)
- 대기 순번 관리 (position)
- 참석 취소 → 대기 1번 자동 승격 (DB 트랜잭션 또는 Edge Function)
- 참석/대기/취소 UI (버튼 상태 전환)
- 참석자/대기자 명단 표시
- 일정 자동 마감 (참석자 = 정원 → closed)
- 참가비 입금 안내 (계좌 표시 + 복사 — 수동)
- Realtime 구독 (참석 현황 실시간 반영)

### 완료 기준
- 캘린더에서 일정 생성 → 카드 표시 → 참석 → 마감 → 대기 → 취소 → 승격 전체 플로우
- 동시 참석 시 race condition 없음 (정원 초과 방지)
- Realtime: 다른 유저가 참석하면 내 화면에 즉시 반영
- 모바일 캘린더 터치 UX 원활

### 기술 고려사항
- 캘린더: 커스텀 구현 (목업 느낌 참고)
- 승격 로직: Edge Function 또는 PostgreSQL RPC (트랜잭션 필수)
- Realtime: `event_attendees` 변경 구독 → React Query 캐시 무효화
- 반응형: 캘린더 (모바일: 컴팩트, 데스크톱: 넓은 그리드)

## Phase 4: 알림 시스템 📋

> **마일스톤**: "일정이 생기면 멤버들이 알림을 받고 앱에 들어온다"

### 목표
푸시 알림으로 사용자를 앱으로 다시 불러온다. 이 플랫폼의 핵심 가치 — "카톡 단톡방 대체".

### 작업 목록
#### Sprint 6: 푸시 알림 + 인앱 알림
- Supabase 마이그레이션: `notifications` 테이블 + RLS
- FCM 설정 (expo-notifications)
- 푸시 토큰 등록 (profiles.push_token)
- 알림 발송 Edge Function:
  - 새 일정 → 전체 멤버
  - 일정 변경/취소 → 참석자+대기자
  - 가입 신청 → Owner+Admin
  - 가입 승인/거절 → 신청자
  - 대기→참석 승격 → 해당 유저
- 딥링크 (알림 탭 → 해당 화면 이동)
- 앱 내 알림 목록 화면 (읽음/안읽음)
- 출석 리마인더 (일정 1시간 전 — pg_cron 또는 스케줄링)
- PWA 웹 푸시 설정 + "홈 화면 추가" 가이드 UI

### 완료 기준
- 일정 생성 → 멤버 푸시 수신 → 탭 → 앱 진입 → 일정 상세 전체 플로우
- 알림 목록 읽음/안읽음 관리
- 리마인더 정시 발송
- Android 웹 + PWA 푸시 정상 동작

### 기술 고려사항
- Edge Function에서 `service_role` 키로 전체 멤버 토큰 조회 + FCM 발송
- 딥링크: `expo-notifications` → `expo-router` 연동
- 스케줄링: Supabase pg_cron 또는 Edge Function Cron
- PWA: iOS 16.4+ "홈 화면 추가" 시 푸시 가능 — 안내 UI 필수

## Phase 5: MVP 완성 + 배포 📋

> **마일스톤**: "실제 동호회가 사용할 수 있는 상태로 웹 배포"

### 목표
기본 통계, 공지사항, 엣지 케이스 처리, QA, 성능 최적화를 거쳐 MVP를 배포한다.

### 작업 목록
#### Sprint 7: 경기 결과 + 통계 + 공지
- Supabase 마이그레이션: `match_results` 테이블 (일정별 경기 결과 기록)
- 경기 결과 입력 UI (일정 종료 후 Owner/Admin이 입력 — 단식/복식, 스코어, 승패)
- 내 전적 요약 (전체/단식/복식 승률, 최근 N경기 W/L)
- 월별 승률 추이 차트
- 회원별 랭킹 (승률 기준, 전체/단식/복식 필터)
- 복식 파트너 궁합 (파트너별 승률, 베스트 궁합)
- 참석 통계 (참석 횟수, 참석률)
- 공지사항 CRUD (Owner/Admin 작성, 멤버 조회)
- 공지 알림 연동

#### Sprint 8: QA + 마무리 + 배포
- 엣지 케이스 처리 (클럽장 탈퇴→이전, 계정 삭제, 동호회 삭제)
- 에러 화면 (네트워크, 404, 권한 없음)
- 빈 상태 화면 (일정 없음, 멤버 없음, 알림 없음)
- 로딩 스켈레톤 UI
- 반응형 전체 점검 (모바일/태블릿/데스크톱 전 화면)
- 성능 최적화 (리스트 가상화, 이미지 최적화, 번들 사이즈)
- SEO 메타 태그 (동호회 링크 공유 시 미리보기)
- Vercel 웹 배포
- 실제 Supabase 프로젝트 생성 + 프로덕션 환경 설정
- 최종 테스트 (실제 사용자 시나리오)

### 완료 기준
- Vercel URL에서 실제 접근 + 전체 플로우 정상
- 경기 결과 입력 → 승률/랭킹 자동 계산 → 통계 화면 표시
- 파트너 궁합 (복식 파트너별 승률) 정상 표시
- 인스타에서 링크 공유 시 미리보기 표시
- 모바일/데스크톱 반응형 전 화면 정상
- Lighthouse 성능 70+

## Phase 8: 다종목 플랫폼 ⏸️

> **마일스톤**: "테니스 외 다른 스포츠 동호회도 사용 가능"

### 목표
다종목 인프라 구축 + 첫 번째 확장 종목 지원. 이후 종목은 Sprint 단위로 추가.

### 작업 목록
#### Sprint 13: 다종목 인프라 + 첫 확장
- `sport_categories` 활성화 (골프, 축구 등 추가)
- 종목 선택 UI (동호회 생성 시)
- 종목별 필터 (동호회 탐색)
- 종목별 특화 필드 (JSONB 확장 — 골프 핸디캡 등)
- 종목별 아이콘/테마
- 종목별 통계 확장 (MVP에서 구축한 통계 시스템을 종목별로 커스터마이징)

#### 이후 종목 추가 (Sprint 단위)
- Sprint N: 골프 동호회 특화 (핸디캡, 스코어카드)
- Sprint N+1: 축구 동호회 특화 (포지션, 팀 구성)
- Sprint N+2: 배드민턴, 탁구, 농구 등

### 기술 고려사항
- `sport_categories` 테이블은 Phase 0에서 이미 설계됨 — 데이터만 추가
- 종목별 특화 필드: JSONB 컬럼으로 유연하게 확장
- 첫 종목 확장에서 인프라를 잡으면, 이후 종목은 데이터 + UI 커스터마이징만
