## 단계별 진행
- 한 스텝씩 진행하고, 사용자와 같이 확인한 뒤 다음으로 넘어간다
- 파일 1개 수정/생성 후 반드시 사용자 확인 대기
- 연관된 파일이라도 가능하면 나눠서 보여주기
- task를 만들어뒀더라도 한꺼번에 달리지 않기
- 대량 작업(8개 파일 동시 등) 절대 금지

**Why:** 여러 파일을 한꺼번에 수정하면 검토 힘들고 방향 수정 시 되돌리기 어려움. 실제로 위반하여 제지당함.

## 코드 변경
- 코드 변경/파일 생성 전에 반드시 내용을 보여주고 확인 받은 후 진행
- 대규모 수정 후에는 사용자에게 확인 요청

## Git 커밋/푸쉬
- 커밋/푸쉬 전에 반드시 먼저 물어보고, 사용자가 승인하면 실행한다
- 핵심: 물어보는 절차를 절대 생략하지 않는다 (과거에 위반 — 반복 금지)
- 빌드가 성공하고 실행까지 문제없는 상태가 확인된 후에만 제안

## Sprint 완료 조건
- Sprint가 끝나면 **사용자가 직접 화면을 확인하고 동작을 테스트**할 수 있는 상태여야 한다
- 빌드 오류 없음 + 실행 가능 + 눈으로 확인 가능한 상태가 최소 조건
- Sprint 완료 후 사용자 리뷰 시간을 반드시 갖는다 (수정 사항 반영)
- 사용자가 "확인 완료"할 때까지 다음 Sprint로 넘어가지 않는다

## 코딩 시 rules 적용
- 코드를 작성하기 **전에** rules/{tech}.md 핵심 규칙을 다시 읽는다
- 코드를 작성하는 **중에** 시니어 개발자처럼 코드 컨벤션을 자연스럽게 적용한다 (하드코딩, 네이밍, 타입, 구조, import 등 전부)
- 코드를 작성한 **후에** rules 원본과 대조하여 위반 사항을 수정한다
- "동작하면 됐지" 마인드 금지 — rules 위반 코드는 동작해도 수정한다

**Why:** 규칙을 만들어놓고 코딩할 때 안 지킨 적 있음. 사용자에게 지적당함. 특정 규칙만 체크하는 게 아니라 전체 코드 품질을 시니어 수준으로 유지해야 함.

## 품질
- 문서/코드 작성 시 항상: 상세하고, 꼼꼼하고, 있어보이게, 베스트 프랙티스로 작성
- 바이브 코딩 시 더 좋은 결과가 나오도록 고민하며 작성 (매번 말하지 않아도 기본 원칙)
